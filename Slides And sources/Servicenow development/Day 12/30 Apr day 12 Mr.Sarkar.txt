ITSM Flow
-incident created (state is new)
-Incident Manager Daily Open Unassigned incident and assign groups and persons to incidents 
-After Assigning the state change to in progress
-SLA Service Level Agreement (Dedicated Timeline upon solving Incident)
-Then the team working contact the customer to have more and more details and add a reason for on hold (and change the state to On hold) and add comment like i need some data
-Then SLA state is paused 
-Then user gives data and state changes to in progress and SLA Resume
-If its a small incident then solve it, if it is not then start creating child incident to divide the tasks to many more incidents and dividing it make it more organized
-After Assigning child incident, then each should have a resolution
-After that, we work on resolving the issue and then make it resolve and give the reason , and resolved doesn't mean its closed.
-After finishing all the child incidents and solve everything then we post a comment asking did it got resolved
-If he responded then we close the incident
-Also the User can cancel the Incident if he solved the problem as well

* Problem Management *

If multiple users raise same incident, maybe also in the same time, then its marked as a problem and dealt with it on another way

Problem Flow:
- Create a problem with the same name of the multiple incident 
- Then Add the incident in the problem 
- Assign the problem to problem analyzer group then state change to Assess
- Then it got confirmed from the problem analyzer as a problem and then state changed to Root Cause analysis
- Then We give more details about the problem given in the root cause analysis by the admin or the one who raised the problem , then change the state to fix in progress 
- Then after that Risk is issue if there is a risk when solving the issue like if solved the system might get restarted
- Then after agreement of the risk the problem will be closed
- After problem is resolved then each incident is updated as resolved and the ID of the problem is copied in the resolved notes

** Then finally incident is solved and the problem as well
** If solving any problem a KB knowledge bas should be done and documented as people can go back and solve the same problem with your KB and then attach the KB to their incident 
** When incident is in the problem then the state of incident will be on Hold and
